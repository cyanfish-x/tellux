# 设计缺陷导致的坑

本文记录 `ViewerOptions.scene` 从扁平配置重构为分组配置时暴露出来的设计问题。核心经验是：公开 API 的结构设计不能只看当前参数数量，要提前考虑领域边界、增长方向和调用扩散成本。

## 背景

早期 `scene` 配置为了实现方便，把不同领域的参数直接拍平在同一层，例如：

```ts
scene: {
  atmosphereAlbedoScale: 1,
  atmosphereSunLightIntensity: 1,
  atmosphereSkyLightIntensity: 1,
  cloudQualityPreset: 'medium',
  cloudCoverage: 0.3,
  surfaceMaterialMode: 'auto'
}
```

当参数还少时，这种写法看起来简单。但随着大气、体积云、地表材质、后处理、调试面板、示例和文档不断增加，扁平结构开始变成一个隐性设计债。

本次重构将公开 API 调整为按领域分组：

```ts
scene: {
  atmosphere: {
    lighting: {
      mode: 'light-source',
      sunLightIntensity: 1,
      skyLightIntensity: 1,
      albedoScale: 1
    }
  },
  clouds: {
    quality: 'medium',
    coverage: 0.3
  },
  surface: {
    materialMode: 'auto'
  }
}
```

## 现象

一次看似只是“把 `scene.cloudQualityPreset` 改成 `scene.clouds.quality`”的 API 调整，最终影响了大量文件：

- `src/types.ts` 的公开配置类型。
- `src/Scene.ts` 的运行时控制入口。
- `src/Viewer.ts` 的默认值解析和模块装配。
- 后处理、大气、云、地表材质的同步逻辑。
- 内置调试面板和示例设置面板。
- README、guide、API 文档和多个 examples。

这说明问题不是某个字段命名不优雅，而是公开 API 的领域边界一开始没有立住，导致扁平字段被复制到越来越多的调用点。

## 根因

1. 把“实现时方便”误当成了“API 简洁”。

   扁平字段短期写起来少一层对象，但它掩盖了参数所属领域。随着参数增加，用户很难判断哪些参数互相关联，维护者也很难保持命名一致。

2. 缺少公开 API 的增长模型。

   大气不是一个单点开关，它天然会继续长出 lighting、scattering、sky、shadow、fallbackAmbientLight 等子领域。体积云也会继续长出 quality、coverage、speed、layer 等子配置。早期没有按这种增长方向建模，后面只能靠字段名前缀补救。

3. 配置结构和运行时结构没有优先统一。

   理想状态下，初始化配置和运行时控制入口应共享同一套心智模型：

   ```ts
   scene.clouds.quality
   viewer.scene.clouds.quality
   ```

   如果初始化是扁平字段，运行时又逐渐需要分组对象，API 会变得难学、难查、难迁移。

4. 没有及时阻止“前缀式命名”继续蔓延。

   `atmosphereSunLightIntensity`、`atmosphereSkyLightIntensity`、`atmosphereAlbedoScale` 这类名字一旦出现多个，就已经是领域对象缺失的信号。继续添加同类字段会让后续重构成本指数级增加。

## 当前修复策略

1. 公开 `ViewerOptions.scene` 改为按领域分组。

   - `scene.atmosphere`
   - `scene.atmosphere.lighting`
   - `scene.atmosphere.scattering`
   - `scene.atmosphere.sky`
   - `scene.atmosphere.shadow`
   - `scene.atmosphere.fallbackAmbientLight`
   - `scene.clouds`
   - `scene.surface`
   - `scene.postProcess`

2. 运行时 `viewer.scene` 控制入口保持同样分组。

   配置路径和运行时路径尽量一致，降低用户记忆成本。

3. 不保留旧公开 API 兼容层。

   当前项目还在快速迭代期，没有外部用户负担。此时应优先修正 API 形状，而不是为了短期兼容继续携带设计债。

4. 文档和示例同步迁移。

   公开 API 的示例是事实上的设计约束。只改类型不改文档，会让旧设计继续通过复制示例传播。

## 后续设计准则

- 新增公开配置前，先判断它属于哪个领域对象；不要为了少写一层对象直接加前缀字段。
- 当同一前缀字段出现第二个或第三个时，优先抽出分组对象，而不是继续命名补丁。
- 初始化配置和运行时控制入口应尽量保持同构，例如 `scene.clouds.quality` 对应 `viewer.scene.clouds.quality`。
- 对可能增长的能力，先设计稳定的领域边界，再填具体参数。
- 对外 API 不要直接反映内部实现步骤；它应该表达用户理解的领域概念。
- 快速迭代期发现公开 API 形状不对，要尽早破坏式修正。等示例、文档、面板、插件和用户代码扩散后，改动成本会远高于早期重构。

## 设计评审清单

新增或调整公开 API 前，至少问一遍：

1. 这个字段属于哪个领域对象？
2. 未来同领域还会增加哪些参数？
3. 初始化配置和运行时 API 是否能使用同一套路径？
4. 字段名是否靠前缀弥补缺失的对象边界？
5. 这个 API 会不会扩散到 examples、docs、widgets、storage、dist 类型声明？
6. 如果半年后破坏式修改，它会影响多少文件？
