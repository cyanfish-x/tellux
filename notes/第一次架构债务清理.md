# 第一次架构债务清理

本文记录第一次面向软件工程、面向对象、Clean Code、长期可维护性和最佳实践的架构债务评审结果。目标不是一次性重写，而是为后续迭代提供稳定的优先级和拆分方向。

## 优先级说明

- P1：高风险债务。容易产生隐性 bug，或显著放大后续功能变更半径，应优先处理。
- P2：中风险债务。主要影响可测试性、API 边界、包结构和长期演进效率。
- P3：低风险债务。当前可接受，但需要记录约束和补充防护。

## 状态说明

- [x] 已完成：对应债务的核心边界已经完成第一轮清理，后续只剩常规维护或可选增强。
- [~] 部分完成：已经有明确拆分或隔离，但仍存在本文描述的主要剩余风险。
- [ ] 未完成：当前代码形态仍基本符合本文原始问题描述。

## 当前状态汇总

- [x] `Scene` 与云层 effect 双状态风险：已通过 `CloudRuntimeState` 和 `AtmosphereManager.applyCloudsState()` 清理。
- [x] `Camera` 与 `TilesetManager` 的 `camera.userData` 隐式协议：已改为显式 `CameraEllipsoidProvider`。
- [x] `types.ts` 过大：已拆成 `src/types/*` 领域类型文件，`src/types.ts` 仅保留聚合出口。
- [~] `TilesetManager` God Manager：surface / terrain / imagery / sampling pool / model plugin 已拆出，剩余 scene tileset registry 等协调职责。
- [~] 高度采样内部状态依赖：已拆出 `HeightSamplingBatcher` 和 `TilesetSamplingAdapter`，但 scheduler / readiness probe / raycast sampler 仍未完全独立。
- [ ] `Viewer` 组合根和行为中心：仍未拆 `WidgetManager`、`ModelLayerManager`、options resolver、render loop。
- [ ] 包入口和公开 API 边界：`src/index.ts` 仍通过 `./Viewer` 中转大量公开类型。

- [ ] widgets 静态引入：`Viewer` 仍静态 import 调试 widgets。
- [ ] shader/string patch 版本防护：patch 仍主要集中在 `AtmosphereManager` 内部。

## [ ] P1：Viewer 仍是过大的组合根和行为中心

处理状态：未完成。`Viewer` 仍直接持有 widgets、model layer map、render loop、destroy 编排、options resolver 和采样调度；当前文件仍是主要膨胀点。

相关文件：

- `src/Viewer.ts`

现状：

- `Viewer` 构造函数同时装配 renderer、scene、tileset、controls、widgets、采样、模型、事件和资源加载。
- `render()` 串起 resize、clock、debug panel、timeline、fallback light、post-processing、tileset、atmosphere、model layer 和高度采样调度。
- `destroy()` 负责释放几乎所有子系统资源。

问题：

- `Viewer` 既是公开门面，又是组合根、生命周期管理器、渲染循环、事件系统和部分业务协调器。
- 任一子系统变更都容易触碰 `Viewer`，主文件继续膨胀。
- 单元测试难以隔离，只能通过集成式方式覆盖。

建议方向：

- 拆出 `ViewerOptionsResolver`，集中处理默认参数和配置归一化。
- 拆出 `RenderLoop` 或 `ViewerRuntimeScheduler`，负责默认 animation loop、每帧调度和采样推进。
- 拆出 `WidgetManager`，负责 DebugSettingsPanel、Timeline 的创建、更新和销毁。
- 拆出 `ModelLayerManager`，负责模型图层 ID、加载、更新和释放。
- `Viewer` 保留公开 API、组合编排和跨模块协调。

## [x] P1：Scene 与云层 effect 仍存在双状态风险

处理状态：已完成第一轮清理。`Scene` 不再直接依赖 `CloudsEffect`，云层用户态通过 `CloudRuntimeState` 交给 `AtmosphereManager.applyCloudsState()` 适配到底层 effect。

相关文件：

- `src/Scene.ts`
- `src/rendering/AtmosphereManager.ts`

现状：

- 大气状态已经走 `Scene -> AtmosphereRuntimeState -> AtmosphereManager.applyAtmosphereState()`。
- 云层状态仍由 `CloudSceneControls` 直接读取 `CloudsEffect` 并写入底层属性，例如 `qualityPreset`、`coverage`、`localWeatherVelocity` 和 cloud layer 高度。

问题：

- 同一个 `Scene` 文件里同时存在两套同步风格：大气使用完整状态快照，云层直接写底层 effect。
- `Scene` 再次知道了底层 effect 的具体属性，公开控制层和渲染 adapter 边界不一致。
- 后续新增云参数时容易回到“用户态与底层态双状态”的老问题。

建议方向：

- [x] 引入 `CloudRuntimeState`。
- [x] `CloudSceneControls` 只维护用户态并输出完整快照。
- [x] `AtmosphereManager` 负责把 `CloudRuntimeState` 适配到 `CloudsEffect`。
- [x] 保持初始化配置和运行时路径同构，例如 `viewer.scene.clouds.*` 只表达用户态。

## [~] P1：TilesetManager 已成为新的 God Manager

处理状态：部分完成。`TilesetManager` 保留 tileset 生命周期、热切换、图层同步和跨模块协调；surface / terrain / imagery overlay / 采样池 / tileset 材质与 glTF 兼容补丁已拆到独立模块。`SceneTilesetRegistry` 仍是后续进一步拆分项。

相关文件：

- `src/tiles/TilesetManager.ts`
- `src/tiles/SurfaceTilesetFactory.ts`
- `src/tiles/TerrainTilesetFactory.ts`
- `src/tiles/ImageryOverlayFactory.ts`
- `src/tiles/HeightSamplingTilesetPool.ts`
- `src/tiles/TilesetModelPlugins.ts`

现状：

- `TilesetManager` 同时负责 surface tileset、terrain tileset、scene 3D Tiles、imagery overlay、terrain provider、采样 tileset pool、图层同步和三方库补丁。
- 文件接近 1000 行，并且已经包含多个不同领域的工厂逻辑。

问题：

- 数据源扩展、地形能力扩展和采样优化都会修改同一大类。
- `createImageryOverlay()`、`createTerrainTileset()`、`createSurfaceTileset()`、`createHeightSamplingTilesets()` 代表不同职责，但共享在同一对象里。
- 三方库兼容补丁散落在 manager 内部，难以独立测试。

建议方向：

- [x] 拆出 `SurfaceTilesetFactory`：负责裸球 surface 和 `GeneratedSurfacePlugin`。
- [x] 拆出 `TerrainTilesetFactory`：负责 quantized-mesh terrain、terrain URL、terrain provider 和 terrain imagery。
- [x] 拆出 `ImageryOverlayFactory`：负责 xyz、Cesium Ion、MVT、WMS、GeoJSON 到 overlay/plugin 的转换。
- [ ] 拆出 `SceneTilesetRegistry`：负责外部 3D Tiles 图层注册、显示、移除和材质模式同步。
- [x] 拆出 `HeightSamplingTilesetPool`：负责采样专用 tileset 的创建、复用、失效和释放。
- [~] 将 WMS release guard、texture fallback、region visibility guard 等补丁收敛到明确的 adapter/helper，并补测试。

## [~] P1：高度采样依赖 3d-tiles-renderer 内部状态过深

处理状态：部分完成。批次划分已拆到 `HeightSamplingBatcher`，对 `3d-tiles-renderer` 非公开运行时字段的读取和 raycast 已集中到 `TilesetSamplingAdapter`；但调度、readiness probe 命名边界、raycast sampler 和 fixture 测试仍未完成。

相关文件：

- `src/sampling/HeightSampler.ts`
- `src/tiles/TilesetManager.ts`
- `src/sampling/HeightSamplingBatcher.ts`
- `src/tiles/TilesetSamplingAdapter.ts`

现状：

- `TilesetSamplingAdapter` 仍自定义了 `SamplingTile`、`SamplingTilesRenderer` 等内部结构类型，用于集中隔离上游非公开字段。
- 采样 readiness 仍读取 `root`、`activeTiles`、`visibleTiles`、`loadingTiles`、`downloadQueue`、`parseQueue`、`processNodeQueue` 等内部字段，但读取点已从 `HeightSampler` 收口到 adapter。
- 同一个类内同时包含任务队列、批次划分、offscreen camera、LoadRegionPlugin、raycast、debug log 和 terrain direct fallback。

问题：

- 对上游 `3d-tiles-renderer` 内部字段高度敏感，版本升级风险高。
- readiness 判断、调度策略和 raycast 策略仍有耦合，难以独立验证。
- 当前 `DEFAULT_SAMPLE_HEIGHT_MOST_DETAILED_RAYCAST_TIME_SLICE_MS` 为 `Infinity`，实际 raycast 阶段不会主动时间分片，和调度策略需要明确取舍。

建议方向：

- [ ] 拆出 `HeightSamplingScheduler`：负责 job 队列、batch 推进和公平调度。
- [x] 拆出 `HeightSamplingBatcher`：负责经纬度排序、分组和区域跨度约束。
- [~] 拆出 `TilesetReadinessProbe`：当前读取已集中到 `TilesetSamplingAdapter`，但 readiness 语义还没有独立 probe 边界。
- [~] 拆出 `TilesetRaycastSampler`：当前 raycast 已集中到 `TilesetSamplingAdapter`，但仍与 readiness adapter 共处一类。
- [x] 保留 `QuantizedMeshTerrainSampler` 作为 direct terrain sampler，并通过更明确的策略对象组合。
- [ ] 为 readiness probe 和 batcher 建立 fixture 测试。

## [x] P1：Camera 与 TilesetManager 通过 camera.userData 隐式耦合

处理状态：已完成第一轮清理。`Camera` 改为通过显式 `CameraEllipsoidProvider` 获取当前活动椭球，`Viewer` 作为组合根负责注入 `tilesets.tileset.ellipsoid`，`TilesetManager` 不再写入 `camera.userData.tilesRenderer`。

相关文件：

- `src/Camera.ts`
- `src/tiles/TilesetManager.ts`

现状：

- `TilesetManager` 把当前活动 tileset 写入 `camera.userData.tilesRenderer`。
- `Camera` 再从 `threeCamera.userData.tilesRenderer` 读取 ellipsoid，用于经纬高视角计算。

问题：

- 这是隐藏协议，不具备类型约束。
- `userData` 是 Three.js 给用户和插件存放任意数据的通道，存在命名冲突风险。
- `Camera` 表面上只依赖 `THREE.PerspectiveCamera`，实际隐式依赖 `TilesRenderer`。

建议方向：

- [x] 引入 `CameraEllipsoidProvider`。
- [x] `Camera` 构造时接收 `getActiveEllipsoid()`。
- [x] 避免通过 `userData` 建立内部模块协议。

## [ ] P2：包入口和公开 API 边界不够干净

相关文件：

- `src/index.ts`

- `src/types.ts`

现状：

- `src/index.ts` 从 `./Viewer` 导入和导出大量符号。

- `src/types.ts` type-only import 了 `Viewer`，`Viewer.ts` 又 import 了大量 public types。

问题：

建议方向：

- `src/Viewer.ts` 只导出 `Viewer` 和极少必要局部类型。
- 其他公开类型从领域类型文件或 `src/types.ts` 聚合导出。
- 避免通过实现文件中转公开 API。

## [ ] P3：三方库 shader/string patch 缺少版本防护

处理状态：未完成。shader patch 仍集中在 `AtmosphereManager` 内部，通过字符串替换和 console warning 降级。

相关文件：

- `src/rendering/AtmosphereManager.ts`

现状：

- `AtmosphereManager` 通过字符串替换 patch `AerialPerspectiveEffect` fragment shader，以增加 inscatter 强度和 horizon blend 控制。

问题：

- patch 依赖上游 shader 文本片段。
- 上游升级后可能静默失效，当前主要通过 console warning 暴露。

建议方向：

- 为 patch helper 增加版本/特征注释，说明依赖的上游 shader hook。
- 将 patch 逻辑拆成独立函数，方便 snapshot 测试。
- 失败时保持可预期降级，并在 debug 模式下暴露状态。

## 建议执行顺序

1. P1：继续高度采样拆分，优先把 `HeightSamplingScheduler` 独立出来，再把 `TilesetSamplingAdapter` 拆成 readiness probe 与 raycast sampler；同步补 batcher / readiness fixture 测试。
2. P1：继续拆 `Viewer`，优先顺序为 `WidgetManager`、`ModelLayerManager`、`ViewerOptionsResolver`、`RenderLoop`。这样能先降低主文件膨胀和静态 widget 耦合。
3. P1/P2：从 `TilesetManager` 拆出 `SceneTilesetRegistry`，收口外部 3D Tiles 图层注册、显示、移除和材质模式同步。
4. P2：引入 Vitest 等轻量测试脚手架，先覆盖 WMS URL 转换、terrain URL normalize、LayerManager、Scene runtime state、HeightSamplingBatcher / readiness probe。
5. P2：整理 public barrel，让 `src/index.ts` 成为唯一公开聚合出口，`Viewer.ts` 回到实现文件角色。
6. P2：处理 widgets 包边界。短期通过 `WidgetManager` 隔离，之后再评估 `tellux/widgets` 子入口或 lazy import。
7. P3：拆出 shader patch helper，补上上游 hook 注释、失败状态和 snapshot 测试。

## 非目标

- 不建议一次性大规模重写。
- 不建议在缺少测试保护的情况下同时调整公开 API、渲染管线和采样策略。
- 不建议把短期修补继续堆在 `Viewer.ts`、`TilesetManager.ts` 或 `HeightSampler.ts` 中。
